# WHY ZYNG

Zyng は「LLM 時代のテキスト」を、もう一度ちゃんと設計し直したい、というところから始まった小さな実験です。

## 1. なぜ Zyng が必要だと思ったか

LLM を使った開発や日常のメモで、だんだんこういう不満がたまっていきました。

- README とコードとプロンプトがバラバラの場所に散らばる
- YAML / JSON / Markdown / Python が混在して、  
  「人間が読む層」がどんどんカオスになる
- ログを残しても、
  - いつの話なのか
  - どの設定・モデルのもとで生成されたのか
  - それが「ログ用」なのか「学習用」なのか「テスト用」なのか
  が後からよくわからない

人間の言葉と、LLM の入出力と、設定とログが「ごちゃ混ぜのテキスト」として積み上がっていく。  
そのカオスを、あとから読み返したり再現したりするのがどんどんつらくなる。

Zyng は、このカオスを**「きれいな一枚のテキスト」に戻すための、小さな言語**です。

## 2. Timeline-first という発想

Zyng の一番の特徴は、**time（いつの話か）**を最初から一級市民にしていることです。

- `:::now`
- `:::yest`
- `:::tomo`
- `:::at:"2025-12-03T21:00:00+09:00"`

のように、「この行はいつのことを語っているか」を、テキスト側に露出させます。

Zyng の設計では、

> 何があったか よりも先に  
> いつ／どのタイミングの話か

を意識する、という順番を意図的に採用しています。

LLM のログやプロンプト、設定を timeline-first に書けるようにすることで、

- 後から読み返したときに意味が追いやすい
- どの時点の知識・前提で生成されたかが辿りやすい
- 「時系列ごとにモデルや方針が変わっていく」ことを素直に表現できる

という利点を狙っています。

## 3. time / role / kind / use という4つの軸

長期的には、Zyng の行には次の 4 つのメタ情報を付けられるようにしたいと考えています。

- **time** – いつの話か
  - `now`, `yest`, `tomo`, `:::at:"..."`
- **role** – 誰の視点か / どのエージェントか
  - 例: `:::role:"self"`, `:::role:"planner-ai"`
- **kind** – これは何の種類の記述か
  - 例: `fact`, `feeling`, `plan`, `policy`
- **use** – この行を何に使うつもりか
  - 例: `log`, `train`, `test`, `policy`

たとえば、将来的にはこういう行を書きたいと考えています。

```zyng
show "User asked about travel to Osaka."
  :::now
  :::role:"user"
  :::kind:"fact"
  :::use:"log"
